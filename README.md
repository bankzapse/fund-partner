# พันธมิตรเงินทุน

ระบบบริหารลูกหนี้ สัญญา รับชำระ รียอด รายรับ-รายจ่าย และรายงาน
พัฒนาตามเอกสาร **SRS: พันธมิตรเงินทุน — เอกสารข้อกำหนดและขอบเขตการพัฒนาโปรแกรม** ครบทั้ง 20 หัวข้อ

---

## เริ่มใช้งาน

```bash
npm install
npm run reset      # สร้างฐานข้อมูลใหม่ + ผู้ใช้ตั้งต้น + ข้อมูลตัวอย่าง
npm start          # เปิดที่ http://localhost:3000
```

ไม่ต้องติดตั้ง PostgreSQL — ตอนรันในเครื่องระบบใช้ **PGlite** (PostgreSQL จริงคอมไพล์เป็น WASM)
เมื่อขึ้น production เพียงตั้ง `DATABASE_URL` ชี้ไป Supabase ก็สลับไปใช้ Postgres ตัวจริงทันที
โดย SQL ชุดเดียวกันทั้งหมด

คำสั่งอื่น

| คำสั่ง | ความหมาย |
|---|---|
| `npm run seed` | สร้างผู้ใช้ตั้งต้นเท่านั้น (ไม่ลบข้อมูลเดิม ไม่ใส่ข้อมูลตัวอย่าง) |
| `npm run dev` | รันแบบ auto-reload |
| `npm test` | รันชุดทดสอบตามเกณฑ์รับมอบงาน (ข้อ 18) |

## หลักการที่ใช้ตัดสินใจ (สำคัญต่อความถูกต้องของตัวเลข)

### 1. จำนวนเงินเก็บเป็น "สตางค์" แบบจำนวนเต็มเสมอ
ทุกคอลัมน์เงินเป็น `BIGINT` หน่วยสตางค์ ไม่มีการใช้ Float ที่ใดเลย (ข้อ 19)
API รับ-ส่งเป็นสตางค์ หน้าเว็บแปลงเป็นบาทตอนแสดงผลเท่านั้น

### 2. ตัดดอกเบี้ยก่อน แล้วจึงตัดเงินต้น
เป็นกติกาเดียวที่ใช้ทั้งระบบ (`allocatePayment` ใน `domain/payments.js`)
ยอดที่จ่ายเกินงวดปัจจุบันจะไหลไปงวดถัดไปโดยอัตโนมัติ
ถ้าต้องการให้ส่วนเกิน "ตัดเงินต้น" แทน (ชำระต้นของดอกลอย / ปิดสัญญาก่อนกำหนด)
ให้ติ๊กตัวเลือกในหน้ารับชำระ

### 3. การบันทึกบัญชีตอนสร้างสัญญา — ใช้วิธียอดเต็ม (gross)
ตัวอย่างสัญญา 1,000 / ค่าทำเอกสาร 100 / งวดแรก 50:

| รายการ | บันทึกเป็น | จำนวน |
|---|---|---|
| เงินปล่อยใหม่ | รายจ่าย (เงินทุน) | 1,000 |
| ค่าทำเอกสาร | รายรับแยกประเภท | 100 |
| งวดแรก | รายการรับชำระจริง (แยกต้น/ดอก) | 50 |
| **เงินสดสุทธิที่ออกจริง** | | **850** |

`contracts.cash_disbursed` เก็บ 850 ไว้แสดงให้ลูกค้าและผู้ใช้เห็นแยกจากเงินต้นตามสัญญา
วิธีนี้ทำให้ทั้งกระแสเงินสด รายได้ และเงินต้นคงเหลือ กระทบยอดตรงกันทั้งหมด

### 4. เงินต้นไม่ใช่รายได้ และเงินปล่อยใหม่ไม่ใช่ค่าใช้จ่าย
ตามเกณฑ์รับมอบงานข้อ 18:

```
รายได้จริง   = ดอกเบี้ย + ค่าทำเอกสาร + รายได้อื่น
กำไรสุทธิ    = รายได้จริง - ค่าใช้จ่ายดำเนินงาน
เงินสดสุทธิ  = เงินรับทั้งหมด - เงินจ่ายทั้งหมด
```

หมวดที่ถูกกันออกจากการคำนวณกำไร เพราะเป็นการหมุนเวียนของเงินทุน:
`เงินปล่อยใหม่/เงินสดจ่ายให้ลูกค้า`, `ถอนเงินทุน/เงินปันผลเจ้าของ`, `เงินทุนที่เจ้าของนำเข้า`

> **ส่วนที่เพิ่มเติมจาก SRS:** เอกสารต้นฉบับไม่ได้ระบุวิธีบันทึก "เงินทุนที่เจ้าของนำเข้า"
> แต่ Dashboard ข้อ 5 ต้องแสดง "เงินทุนทั้งหมด" จึงเพิ่มปุ่ม **± เงินทุนเจ้าของ**
> ในหน้ารายรับ-รายจ่าย โดยบันทึกเป็นการเคลื่อนไหวของทุน ไม่กระทบกำไร

### 5. การรียอด — ไม่แสดงว่าจ่ายเงินสดซ้ำ
`ยอดสัญญาใหม่ = เงินต้นคงเหลือเดิม + เงินเพิ่มใหม่`
สัญญาเดิมเปลี่ยนสถานะเป็น `ปิดด้วยการรียอด` (ไม่ลบข้อมูล) และผูกกันผ่านตาราง `contract_links`
เงินสดที่ลูกค้าได้รับคำนวณจาก **เงินเพิ่มใหม่** เท่านั้น (ปรับเป็นยอดเต็มได้ที่หน้าตั้งค่า)

### 6. ไม่ลบข้อมูลการเงินถาวร
รายการรับเงิน รายรับ และรายจ่าย ใช้การ **ยกเลิก** (`is_void`) พร้อมเหตุผลและผู้ยกเลิก
ทุกการสร้าง/แก้ไข/ยกเลิก/อนุมัติ ถูกบันทึกลง `audit_logs` พร้อมค่าเดิมและค่าใหม่

### 7. ตารางงวดกับเงินต้นอาจไม่เท่ากัน — ระบบจะเตือน ไม่แก้ตัวเลขให้เอง
ธุรกิจกำหนดค่างวดและดอกเบี้ยเองได้อิสระตาม SRS ข้อ 7.1
ถ้าผลรวมการตัดเงินต้นตามตารางน้อยกว่าเงินต้นตามสัญญา ระบบจะแสดงคำเตือนพร้อมส่วนต่าง
ก่อนยืนยัน — เงินต้นที่เหลือหลังครบงวดคือยอดที่ใช้ในการรียอด

### ข้อ 19 — ข้อเสนอแนะด้านเทคนิค

- ✅ Web Application แบบ Responsive + `manifest.webmanifest` (ติดตั้งเป็น PWA ได้)

- ✅ วันที่/เวลาอิงโซนเวลา `Asia/Bangkok`

- ✅ ป้องกันรายการซ้ำเมื่อใช้งานหลายคนพร้อมกัน — เลขที่เอกสารออกจากตาราง `counters` ภายใน transaction

## นำเข้าข้อมูลเดิมจาก Excel

เมนู **นำเข้าข้อมูล** (เจ้าของเท่านั้น) รองรับ `.xlsx` และ `.csv` ทำงาน 4 ขั้น

1. **เลือกไฟล์** — เลือกว่าจะนำเข้า "ลูกหนี้ + สัญญา" หรือ "ลูกหนี้อย่างเดียว"
2. **จับคู่คอลัมน์** — ระบบเดาให้จากหัวตารางภาษาไทย แก้เองได้ เลือกชีตและแถวหัวตารางได้
3. **ตรวจสอบ** — ตรวจทุกแถวโดยยังไม่บันทึกอะไร แสดงข้อผิดพลาดรายแถว
4. **นำเข้า** — ทำใน transaction เดียว แถวที่ผิดถูกข้าม แถวที่ถูกต้องเข้าระบบ

### สิ่งที่ระบบอ่านให้อัตโนมัติ

| เรื่อง | รองรับ |
|---|---|
| หัวคอลัมน์ | ไทยและอังกฤษหลายแบบ เช่น `เงินต้น` `ยอดจัด` `วงเงิน` `principal` |
| จำนวนเงิน | `1,000.50` · `฿2,000` · เว้นวรรค |
| วันที่ | `2026-07-20` · `20/07/2569` (พ.ศ. แปลงเป็น ค.ศ. ให้) · serial ของ Excel |
| ประเภทสัญญา | `รายวัน` `รายเดือน` `ดอกลอย` `daily` `monthly` `floating` |
| หัวรายงานด้านบน | ข้ามให้อัตโนมัติ หาแถวหัวตารางเอง |

### หลักการสำคัญ — ข้อมูลนำเข้าคือ "ยอดยกมา"

สัญญาที่นำเข้าคือสัญญาที่เดินอยู่แล้วก่อนใช้ระบบนี้ ระบบจึง **ไม่สร้าง**

- รายรับค่าทำเอกสาร
- รายจ่ายเงินปล่อยใหม่
- รายการรับชำระงวดแรก

เพราะเงินก้อนนั้นเคลื่อนไหวไปก่อนแล้ว ถ้าบันทึกซ้ำ **กำไรและกระแสเงินสดของงวดปัจจุบันจะผิด**
สิ่งที่สร้างคือ ลูกหนี้ + สัญญา + ตารางงวด + เครื่องหมายว่างวดไหนชำระไปแล้ว
ทำให้รับชำระงวดถัดไปได้ทันทีโดยตัดจากยอดยกมาอย่างถูกต้อง

## การนำขึ้นใช้งานจริง (Vercel + Supabase)

### สถานะปัจจุบัน

| ส่วน | สถานะ |
|---|---|
| Vercel | ✅ deploy แล้ว — https://fund-partner.vercel.app (ทีม chao-dee) |
| หน้า SEO `/welcome` | ✅ เปิดสาธารณะ Google เข้าถึงได้ |
| ตัวแอป (login / API) | ⚠️ ตอบ 503 จนกว่าจะตั้ง `DATABASE_URL` |
| Supabase | ❌ ยังไม่ได้สร้าง — ต้องทำตามขั้นตอนด้านล่าง |

> ระบบตั้งใจให้ตอบ 503 พร้อมข้อความบอกวิธีแก้ แทนที่จะใช้ที่เก็บข้อมูลชั่วคราวบน Serverless
> ซึ่งจะทำให้ข้อมูลการเงินหายโดยไม่มีสัญญาณเตือน

### วิธีที่ 1 — สคริปต์อัตโนมัติ (แนะนำ)

```bash
bash scripts/setup-supabase.sh
```

สคริปต์จะสร้างฐานข้อมูล ดึงตัวแปร สร้างตาราง สร้างผู้ใช้ตั้งต้น
สร้าง Storage bucket deploy ใหม่ และตรวจสอบให้ครบในคำสั่งเดียว

**ต้องทำครั้งเดียวก่อน:** กดยอมรับข้อตกลงของ Supabase ที่
<https://vercel.com/chao-dee/~/integrations/accept-terms/supabase?source=cli>
(เป็นการยอมรับ EULA และ Privacy Policy ในนามทีม จึงต้องเป็นเจ้าของบัญชีกดเอง)

### วิธีที่ 2 — สร้างที่ supabase.com เอง แล้วค่อยตั้งค่า

ถ้าไม่อยากผ่าน Vercel Marketplace สร้างโปรเจกต์ที่ supabase.com โดยตรง
คัดลอก connection string แบบ **Connection pooling** (พอร์ต 6543) แล้ว

```bash
npx vercel env add DATABASE_URL production      # วาง connection string
DATABASE_URL='<connection string>' npm run seed # สร้างตาราง + ผู้ใช้
npx vercel --prod                               # deploy ใหม่
```

### รายละเอียดแต่ละขั้น (ถ้าอยากทำเอง)

### 1. สร้างฐานข้อมูลบน Supabase

1. สร้างโปรเจกต์ใหม่ที่ [supabase.com](https://supabase.com) — เลือก region **Southeast Asia (Singapore)** เพื่อให้ใกล้ไทยที่สุด
2. ไปที่ **SQL Editor → New query** วางเนื้อหาไฟล์ `supabase/migrations/0001_init.sql` แล้วกด Run
   (หรือข้ามขั้นนี้ก็ได้ ระบบจะสร้างตารางให้เองเมื่อเชื่อมต่อครั้งแรก)
3. ไปที่ **Project Settings → Database → Connection string → URI**
   เลือกแบบ **Connection pooling** (พอร์ต 6543) แล้วคัดลอกไว้ — Serverless ต้องใช้แบบ pooling
4. ไปที่ **Storage → New bucket** ตั้งชื่อ `fund-partner` และตั้งเป็น **Public** (สำหรับเปิดดูรูปใบเสร็จ)

### 2. Deploy ขึ้น Vercel

```bash
npm i -g vercel
vercel login
vercel link          # เชื่อมกับ repo bankzapse/fund-partner

# ตั้งค่า environment variables
vercel env add DATABASE_URL production               # วาง connection string จากข้อ 1.3
vercel env add SUPABASE_URL production               # https://xxxx.supabase.co
vercel env add SUPABASE_SERVICE_ROLE_KEY production  # Project Settings > API > service_role
vercel env add SUPABASE_STORAGE_BUCKET production    # fund-partner
vercel env add NODE_ENV production                   # production

vercel --prod
```

### 3. สร้างผู้ใช้ชุดแรกบนฐานข้อมูลจริง

```bash
DATABASE_URL='<connection string จาก Supabase>' npm run seed
```

> ใช้ `npm run seed` (ไม่ใช่ `reset`) เพื่อสร้างเฉพาะผู้ใช้ตั้งต้น ไม่ใส่ข้อมูลตัวอย่าง
> จากนั้น **เปลี่ยนรหัสผ่านทั้ง 4 บัญชีทันที**

### 4. ตรวจหลัง deploy

| ตรวจ | คาดหวัง |
|---|---|
| `https://<โดเมน>/welcome` | หน้าแนะนำระบบ (SEO) |
| `https://<โดเมน>/` | หน้าเข้าสู่ระบบ |
| เข้าสู่ระบบแล้วสร้างสัญญา | เลขที่สัญญาเดินต่อเนื่อง ไม่ซ้ำ |
| แนบรูปใบเสร็จ | ไฟล์ขึ้นไปอยู่ที่ Supabase Storage |

### หมายเหตุเรื่อง Serverless

- **Connection pooling บังคับใช้** — Serverless เปิด connection ใหม่ได้เร็วมาก ถ้าต่อตรงพอร์ต 5432 จะเต็มโควตา
- **ไฟล์แนบต้องไปที่ Supabase Storage** — ระบบไฟล์ของ Vercel เขียนไม่ได้และหายทุก request
- **การสำรองข้อมูล** — Supabase สำรองให้อัตโนมัติอยู่แล้ว ปุ่มในหน้าตั้งค่าเป็นการส่งออก JSON เพิ่มเติมไว้เก็บเอง
  (ไม่ส่งออก `password_hash` เพื่อความปลอดภัย)
- **`purgeSessions` ถูกปิดบน Serverless** — เพราะ process ไม่อยู่ยาว session ที่หมดอายุถูกปฏิเสธตอนตรวจอยู่แล้ว

## สิ่งที่ต้องทำก่อนเปิดใช้งานจริง

1. **เปลี่ยนรหัสผ่านผู้ใช้ตั้งต้นทั้ง 4 บัญชี** และลบบัญชีที่ไม่ใช้
   — สำคัญมากเพราะรหัสตั้งต้นเปิดเผยอยู่ใน repo สาธารณะ
2. เปิด **HTTPS** และตั้ง `NODE_ENV=production` (ข้อ 15)
3. ตั้ง **การสำรองฐานข้อมูลอัตโนมัติ** — ระบบมีปุ่มสำรองด้วยมือในหน้าตั้งค่า
   แนะนำเพิ่ม cron สำรองไฟล์ `data/` และ `uploads/` รายวัน
4. ตรวจค่าตั้งต้นในหน้า **ตั้งค่า**: ค่าทำเอกสาร, เกณฑ์ค้างชำระ, ฐานคำนวณเงินสดตอนรียอด
5. **ข้อ 20 — ด้านกฎหมาย:** ให้ผู้เชี่ยวชาญด้านกฎหมายและบัญชีตรวจสอบรูปแบบสัญญา
   อัตราดอกเบี้ย ค่าธรรมเนียม การจัดเก็บข้อมูลส่วนบุคคล (PDPA) และเอกสารทางภาษี
   ให้สอดคล้องกับกฎหมายไทยก่อนเปิดใช้งาน
   ระบบนี้เปิดให้ตั้งค่ากติกาทั้งหมดได้ ไม่มีการฝังอัตราดอกเบี้ยหรือค่าธรรมเนียมแบบแก้ไขไม่ได้
